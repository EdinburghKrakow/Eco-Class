# Эко-Вкус — полная спецификация проекта

## 1\. О проекте

**Эко-Вкус** — это веб-приложение интернет-магазина фермерских продуктов. Пользователь может просматривать каталог, открывать карточки товаров, читать описание, добавлять позиции в корзину, оформлять заказ с доставкой или самовывозом, а также отслеживать историю покупок через личный кабинет.

В текущей реализации проект уже включает не только клиентскую часть на **React + Vite**, но и серверную логику на **Node.js + Express** с подключением к **PostgreSQL**. Кроме пользовательского сценария, в приложении есть **административная панель**, где можно:

* добавлять, редактировать и удалять товары;
* просматривать активные заказы;
* менять статус заказа;
* смотреть аналитику продаж и топ продаваемых товаров.

Проект ориентирован на простой и понятный процесс заказа натуральных продуктов: от знакомства с ассортиментом до оформления покупки и контроля заказов со стороны администратора.

## 2\. Философия и пользовательский опыт

### Наш пользователь

**Имя:** Анна, 34 года.  
**Род занятий:** офисный специалист, заботится о качестве питания семьи.  
**Поведение:** заказывает продукты с ноутбука, компьютера или телефона, ценит понятный интерфейс, быстрый выбор товаров и прозрачный процесс оформления заказа. Ей важно видеть наличие, цену и иметь возможность сохранить адрес для следующих покупок.

### Какую проблему она решает?

Анна хочет быстро заказать свежие фермерские продукты без поездки на рынок. Ей нужен сервис, где можно:

* удобно посмотреть ассортимент;
* собрать корзину;
* выбрать способ получения заказа;
* сохранить свои данные для повторных покупок;
* видеть историю оформленных заказов.

### Его окружение

* **Устройства:** ноутбук, планшет, смартфон.
* **Контекст:** оформление заказа вечером дома, быстрый повторный заказ днём, просмотр каталога в перерывах.
* **Ожидания:** минимум лишних шагов, понятная корзина, аккуратный визуал, быстрое взаимодействие.

## 3\. Пользовательские истории (User Stories)

**История 1: Просмотр каталога**

> Как покупатель, я хочу видеть список товаров с изображениями, названием, ценой и доступностью, чтобы быстро выбрать нужные продукты.

**История 2: Изучение товара**

> Как покупатель, я хочу открыть карточку товара и прочитать подробное описание, чтобы понять, подходит ли он мне.

**История 3: Регистрация и вход**

> Как покупатель, я хочу зарегистрироваться и войти в аккаунт по номеру телефона и паролю, чтобы сохранять свои данные и историю заказов.

**История 4: Управление профилем**

> Как покупатель, я хочу сохранить адрес доставки, комментарий и адрес магазина для самовывоза, чтобы не вводить эти данные каждый раз заново.

**История 5: Корзина**

> Как покупатель, я хочу добавлять товары в корзину, изменять количество и удалять позиции, чтобы гибко собирать заказ.

**История 6: Оформление заказа**

> Как покупатель, я хочу выбрать между доставкой и самовывозом, чтобы получить заказ удобным для меня способом.

**История 7: История заказов**

> Как покупатель, я хочу видеть список прошлых заказов со статусами, чтобы понимать, что и когда я заказывал.

**История 8: Работа администратора**

> Как администратор, я хочу управлять товарами, отслеживать незавершённые заказы и менять их статусы, чтобы поддерживать работу магазина.

**История 9: Аналитика**

> Как администратор, я хочу видеть выручку, популярные товары и статистику по способам получения заказа, чтобы понимать эффективность продаж.

## 4\. Технологический стек

### Фронтенд

* **React 19** — построение пользовательского интерфейса
* **Vite** — сборка и dev-сервер
* **JavaScript (ES6+)** — логика приложения
* **CSS3** — стилизация
* **Fetch API** — запросы к серверу

### Бэкенд

* **Node.js** — серверная среда выполнения
* **Express** — REST API
* **PostgreSQL** — база данных
* **pg** — драйвер для работы с PostgreSQL
* **crypto.scrypt** — хеширование паролей
* **cors** — настройка CORS между фронтендом и API

### Архитектурная особенность текущей реализации

В проекте серверная логика расположена в одном файле `server.js`, но фактически запускает **четыре Express-приложения** на разных портах:

* `3001` — товары и статические файлы;
* `3002` — авторизация и профиль;
* `3003` — заказы и аналитика;
* `3004` — корзина.

## 5\. Архитектура приложения

### Общая схема взаимодействия

```text
React (Vite, порт 5173)
        │
        ├── GET/POST/PUT/DELETE → Goods API (порт 3001)
        ├── POST/GET/PUT        → Auth API (порт 3002)
        ├── GET/POST/PATCH      → Orders API (порт 3003)
        └── GET/POST/PATCH      → Cart API (порт 3004)
                                  │
                                  └── PostgreSQL (EcoTaste)
```

### Особенности текущего фронтенда

* Основная логика собрана в одном крупном компоненте `App.jsx`.
* В приложении нет выделенных React-компонентов по папкам `components/`, вся основная UI-логика сосредоточена в одном месте.
* Данные о товарах, корзине, профиле, заказах и аналитике хранятся в локальном состоянии через `useState`.
* Загрузка данных выполняется через набор функций из `src/api.js`.

## 6\. Основные сценарии работы приложения

### Пользовательская часть

1. Пользователь открывает каталог товаров.
2. Нажимает на карточку товара и получает подробное описание в модальном окне.
3. Регистрируется или входит в аккаунт.
4. Сохраняет данные профиля: адрес доставки, комментарий, адрес магазина для самовывоза.
5. Добавляет товары в корзину.
6. Изменяет количество позиций.
7. Выбирает способ получения заказа:

   * **delivery** — доставка;
   * **pickup** — самовывоз.
8. Оформляет заказ.
9. Смотрит историю заказов и их статусы.

### Административная часть

1. Администратор входит в систему.
2. Переключается в режим админ-панели.
3. Работает с разделами:

   * склад;
   * активные заказы;
   * аналитика продаж.
4. Может:

   * создать новый товар;
   * отредактировать товар;
   * удалить товар;
   * изменить статус заказа;
   * посмотреть сводную статистику продаж.

## 10\. Логика корзины и оформления заказа

### Формат хранения корзины

В текущей реализации корзина хранится в БД не как отдельные строки по товарам, а как две строки с разделителем `;`:

```text
goods\_cart = "1081050;1321354;5465416"
quantity   = "2;1;3"
```

Сервер преобразует эти строки в объект вида:

```javascript
{
  "1081050": 2,
  "1321354": 1,
  "5465416": 3
}
```

### Оформление заказа

При оформлении заказа сервер:

1. проверяет пользователя;
2. определяет адрес в зависимости от `deliveryType`;
3. загружает корзину;
4. проверяет наличие товара на складе;
5. уменьшает остатки;
6. создаёт запись в таблице `orders`;
7. очищает корзину пользователя.

### Статусы заказа

В интерфейсе используются следующие статусы:

* `Собираем`
* `Ищем курьера`
* `В пути`
* `Ожидает выдачи`
* `Завершен`

## 13\. Технические ограничения и зоны для улучшения

### Что видно по текущему коду

1. **Вся фронтенд-логика сосредоточена в одном `App.jsx`** — проект стоит разбить на компоненты.
2. **Серверный код находится в одном `server.js`**, хотя логически уже разделён на несколько сервисов.
3. **Нет JWT / сессий** — авторизация строится на передаче логина, без полноценной защиты API.
4. **Корзина и состав заказа хранятся в строках через `;`**, что усложняет масштабирование и аналитику.
5. **В `package.json` есть `react-router-dom`, но маршрутизация в текущей реализации не является основой архитектуры.**
6. **Нет явного разделения ролей на уровне middleware** — админ-функции завязаны на логике клиента и полях пользователя.
7. **Конфигурация БД захардкожена в коде**, вместо переменных окружения.
8. **В архив включена папка `node\_modules`**, что утяжеляет проект и обычно не хранится в репозитории.

### Что можно улучшить в следующей версии

* вынести конфигурацию в `.env`;
* разделить фронтенд на `components`, `pages`, `hooks`, `services`;
* внедрить JWT-аутентификацию;
* нормализовать БД через таблицы `order\_items` и `cart\_items`;
* добавить protected routes и middleware для администратора;
* реализовать адаптивную вёрстку под мобильные устройства;
* добавить поиск, фильтрацию и категории товаров;
* подключить уведомления об успешных действиях вместо `alert`;
* покрыть API валидацией и тестами.

**Эко-Вкус** в текущем виде — это уже не просто макет интерфейса, а рабочая основа интернет-магазина фермерских продуктов с каталогом, авторизацией, корзиной, оформлением заказа, историей покупок и административным контуром.

Главная ценность проекта — в том, что он покрывает полный цикл взаимодействия между покупателем и магазином. При этом архитектура пока остаётся учебной и монолитной: фронтенд собран вокруг одного большого компонента, а сервер — вокруг одного файла с несколькими Express-приложениями. Это делает проект хорошей базой для следующего шага: рефакторинга, усиления безопасности и подготовки к более серьёзной эксплуатации.

## 16\. Краткий вывод по анализу архива

По присланному архиву видно, что проект **Эко-Вкус** уже реализует:

* витрину товаров;
* работу с карточками продуктов;
* регистрацию и вход;
* профиль пользователя;
* корзину;
* оформление заказов;
* историю заказов;
* администраторский режим;
* аналитику продаж.

То есть README для этого проекта должен описывать не абстрактный будущий магазин, а **уже существующую реализацию интернет-магазина на React + Express + PostgreSQL**, что и отражено в этой версии документа.
